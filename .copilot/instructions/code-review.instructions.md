# 代码审查与安全编码实践

## 代码审查清单

### 1. 功能正确性
- 逻辑是否完整覆盖了需求中的所有场景（正常路径 + 边界 + 异常）？
- 边界条件：空输入、零值、最大值、溢出、下溢
- 返回值和错误码是否全部被检查？
- 状态机转换是否覆盖了所有合法状态 + 非法状态的防御处理？

### 2. 安全性（Security）
- 所有外部输入（CAN 报文、诊断请求、传感器数据）必须验证范围和格式
- 数组/缓冲区操作必须有边界检查，防止溢出
- 禁止使用 `strcpy`/`sprintf`/`gets` 等不安全函数，使用 `strncpy`/`snprintf` 替代
- 密钥/种子/密码不应硬编码在源文件中
- Flash 写入前必须验证地址范围和数据完整性（CRC/签名）
- 诊断安全访问等级 (Security Access) 的种子必须使用真随机数

### 3. 可靠性（Reliability）
- 共享资源（全局变量、外设寄存器）的并发访问是否有保护（临界区/互斥）？
- 中断使能/禁用是否成对出现，不会导致中断永久禁用？
- Watchdog 喂狗是否在所有执行路径中都能及时执行？
- 除零、指针解引用前是否有有效性检查？

### 4. 可维护性
- 魔数应使用 `#define` 或 `enum` 定义并添加注释
- 复杂条件表达式应拆分为命名中间变量
- 函数参数不超过 5 个；超过时考虑使用结构体
- 新增代码是否有对应单元测试或系统测试场景？

### 5. 性能
- 循环中是否有不必要的重复计算？（提取到循环外）
- ISR 执行时间是否评估过？（应控制在微秒级）
- 大数组/结构体是否通过指针传递而非值传递？

## 审查回复格式

审查代码时使用以下格式：
```
**[严重程度]** 行 XX: 简短描述
- 问题：具体说明问题
- 影响：可能导致的后果
- 建议：推荐的修复方式
```

严重程度：🔴 Critical（必须修复）| 🟡 Major（应该修复）| 🔵 Minor（建议修复）| ℹ️ Info（供参考）
