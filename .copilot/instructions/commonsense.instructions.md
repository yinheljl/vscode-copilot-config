# 通用开发最佳实践

## 代码质量原则

- **简单优于复杂**：能用简单方式实现，不要过度抽象
- **早期返回**：条件不满足时尽早 return，避免深层嵌套
- **单一职责**：每个函数只做一件事，函数体控制在 75 行以内
- **函数参数**：不超过 5 个，超过时用结构体封装
- **不可变性优先**：能用 const 的地方全用 const

## 命名规范

- 使用描述性名称，让名字说明意图
- 布尔变量用 `is_/has_/can_` 前缀（如 `is_ready`、`has_error`）
- 避免缩写，除非是领域内公认的（如 `Dem`、`NvM`、`Dcm`）
- 同一概念在整个代码库中使用一致的名称

## 错误处理

- 所有函数返回值必须检查，不得忽略
- 错误信息要有上下文（哪个模块、哪个操作、当前值是什么）
- 失败时快速退出，不要在错误状态下继续执行

## 安全基础

- 永不信任外部输入（CAN 报文、诊断请求、传感器数据）
- 处理外部数据前验证范围和格式
- 密钥/密码/种子不硬编码在源文件中

## 代码整洁

- 删除注释掉的旧代码（用 git 管理历史，不要留在代码里）
- 删除未使用的变量、函数、头文件
- 消除重复代码（DRY 原则），但不要为了 DRY 强行抽象

## 工具调用效率（AI 行为规范）

- 能从已有知识直接回答的，不调用工具搜索
- 相互独立的工具调用必须并行发出，不串行等待
- 读文件前先判断是否真的需要读，避免冗余工具调用
- 批量操作优于多次单独操作
